# 監査レポート：年金試算（ⓒ）

- **対象領域**: ⓒ 公的年金（国民年金・厚生年金）の自動試算
- **監査日**: 2026-04-24
- **信頼度判定**: ❌ 要対応

## 対象範囲

- `_calcPensionCore(employType, koseiYears, avgIncome, kokuminYears)` (`index.html:15115-15127`) — 国民年金・厚生年金の月額を算出するコア関数。純粋関数（DOM 非依存）。
- `calcPensionEstimate(who)` (`index.html:15129-15169`) — `_calcPensionCore` を DOM 入力／出力に連結する UI ラッパー。本人 (`self`) とパートナー (`partner`) の 2 モード。
- `applyPensionEstimate(who)` (`index.html:15171-15182`) — 試算結果を `retPensionMonthly` / `retPensionMonthly_p` に転記。
- 定数 `KOKUMIN_FULL_MONTHLY = 6.8` (`index.html:15100`) / `KOKUMIN_FULL_MONTHS = 480` (`index.html:15101`)。
- 旧 `calcPensionEstimate()`（引数なし、`index.html:5944-5960`）— **死にコード**。後方の新実装（15129 行）と関数名が衝突しており、JavaScript のホイスト上書きで新実装だけが生き残る。旧実装は `calcSimpleSim()` が唯一の呼び出し箇所だが、新実装の署名 `(who)` が引数なし呼び出し時に `'self'` として動作するため、副作用（DOM 描画）付きで呼ばれる。**本監査は新実装（15129 行）を一次対象**とし、旧実装・重複定義問題は §6 で別途扱う。
- 繰上げ・繰下げシミュレーション `calcAndShowDeferralSim()` (`index.html:17790-17888`) — 65 歳基準を入力として、60〜75 歳の各開始年齢の月額・損益分岐年齢を **表示のみ**（シミュレーションへの反映はなし。`state.retirement.pensionMonthly` は繰上げ/繰下げ後の値をユーザー自身が入力する想定）。
- マクロ経済スライド `pensionSlide` — UI で 0% / 10% / 20% の 3 シナリオ（`index.html:3243-3278`）。`calcPensionEstimate` の結果には **反映されない**。`pensionSlide` は統合シミュレーション側で `pensionAnnual * (1 − pensionSlide)` として 1 回だけ適用される（`index.html:15342-15343` 等）。

**呼び出し元**:
- UI 入力変更時のリアクティブ呼び出し: `onchange` / `oninput` で `calcPensionEstimate(who)` が実行される（`index.html:3321, 3329, 3335, 3344, 3376, 3384, 3390, 3399`）。
- シナリオ読込時 (`index.html:14663-14664`): `calcPensionEstimate('self')` / `calcPensionEstimate('partner')` をまとめて再計算。
- タブ切替時 (`index.html:15111`): `switchPenTab('partner')` で初回呼び出し。
- 旧 `calcPensionEstimate()` 呼び出し: `calcSimpleSim` (`index.html:5979`) / `renderTaxSimulation` (`index.html:20374, 20491`) — これらは新実装を `(who=undefined → 'self')` で呼ぶことになり、**副作用として本人の年金試算 UI が再描画される**（想定外の副作用）。

## 1. 関数の目的と入出力

### `_calcPensionCore(employType, koseiYears, avgIncome, kokuminYears)`

- **目的**: 厚生労働省の公的年金計算式に基づく簡易試算。国民年金（基礎年金）と厚生年金（報酬比例部分）を独立に計算し、合計の税込み月額と手取り推計月額を返す。
- **入力**:
  - `employType: 'employee' | 'self' | 'part'`（会社員・公務員 / 自営業・フリーランス / パート）
  - `koseiYears: number`（厚生年金加入年数、`employType === 'employee'` のときのみ有効）
  - `avgIncome: number`（平均税込年収、万円）
  - `kokuminYears: number`（国民年金加入年数、0〜40）
- **戻り値**: `{ kokuminMonthly, koseiMonthly, grossTotal, netTotal, kokuminMonths }`（万円/月、`kokuminMonths` のみ月単位）

### `calcPensionEstimate(who)`

- **目的**: DOM から入力を読み、`_calcPensionCore` を呼び、結果を DOM に描画し `window._pensionEstimate[who] = { net }` に保存。世帯合計も更新。
- **入力**: `who: 'self' | 'partner'`（省略時 `'self'`）
- **戻り値**: `undefined`（副作用のみ）

## 2. 使用している計算式

### 2.1 国民年金（基礎年金）の按分計算（`index.html:15116-15117`）

```js
const kokuminMonths = Math.min(kokuminYears * 12, 480);
const kokuminMonthly = Math.round(KOKUMIN_FULL_MONTHLY * (kokuminMonths / KOKUMIN_FULL_MONTHS) * 10) / 10;
```

- 数式: `月額 = 6.8 万円 × (加入月数 / 480 ヶ月)`、`加入月数 = min(加入年数 × 12, 480)`。
- 40 年満額で月 6.8 万円、30 年で `6.8 × 360/480 = 5.1 万円`、20 年で `6.8 × 240/480 = 3.4 万円`。

### 2.2 厚生年金（報酬比例部分）（`index.html:15118-15123`）

```js
if (employType === 'employee' && koseiYears > 0 && avgIncome > 0) {
  const hyojunGekkyu = Math.min(avgIncome / 12, 65);
  const annualKosei = hyojunGekkyu * (5.481 / 1000) * (koseiYears * 12);
  koseiMonthly = Math.round(annualKosei / 12 * 10) / 10;
}
```

- 数式: `年額 = min(平均年収/12, 65 万円) × (5.481/1000) × (加入年数 × 12)`、`月額 = 年額 / 12`。
- 「平均年収（税込）」は UI ラベル通りの税込年額（万円）。12 で割って月額化し、65 万円/月でキャップ。
- 乗率 **5.481/1000** は 2003 年 4 月以降（総報酬制導入後）の値。**2003 年 3 月以前の 7.125/1000 は実装されていない**（§3.3 で詳述）。

### 2.3 「手取り」概算（`index.html:15124-15125`）

```js
const grossTotal = Math.round((kokuminMonthly + koseiMonthly) * 10) / 10;
const netTotal = Math.round(grossTotal * 0.87 * 10) / 10;
```

- 数式: `手取り = 税引前合計 × 0.87`（13% を税・社保控除として一律差引）。
- UI ラベル「手取り目安」・「税・社会保険料控除後の概算（約87%）」 (`index.html:15150`) と整合。

### 2.4 UI 側の税込年収推計（`index.html:3335-3340`）

- UI には「平均年収（税込み）万円」と明記されており、**税引前年額**を入力する設計。
- 別経路で「手取り年収 ÷ 0.78 で税込みを自動推計」する機能が UI 警告内で言及されるが (`index.html:3337`)、旧 `calcPensionEstimate()` (5944-5960) の `income / 0.75` ロジックの名残で、新実装は DOM 入力の値をそのまま税込として扱う（gross 推計の自動適用はない）。

### 2.5 繰上げ・繰下げ受給（`index.html:17815-17818`）

```js
function adjustRate(startAge) {
  const months = (startAge - BASE_AGE) * 12;
  return months < 0 ? 1 + months * 0.004 : 1 + months * 0.007;
}
```

- 数式: 繰上げ（`startAge < 65`）: `1 + (startAge − 65) × 12 × 0.004`、繰下げ（`startAge > 65`）: `1 + (startAge − 65) × 12 × 0.007`。
- 60 歳受給: `1 + (−60) × 0.004 = 0.76`（−24%）、75 歳受給: `1 + 120 × 0.007 = 1.84`（+84%）。
- 2022 年 4 月改正後の率と整合。
- **この関数の結果は表示専用**（`panel.innerHTML` にテーブル描画のみ）。`_calcPensionCore` の出力や `state.retirement.pensionMonthly` には**自動反映されない**。

## 3. 標準との突合

### 3.1 国民年金（基礎年金）満額

- **標準（2024 年度 / 令和 6 年度）**: 年額 **816,000 円** = 月額 **68,000 円** = **6.8 万円/月**。
  - 出典: 厚生労働省「令和 6 年度の年金額改定について」 <https://www.mhlw.go.jp/stf/newpage_38201.html>
- **標準（2025 年度 / 令和 7 年度）**: 2024 年度から **+1.9%** 改定。年額 `816,000 × 1.019 = 831,504 円` → 公表値 **831,696 円**（= 月額 **69,308 円 ≈ 6.93 万円/月**）。
  - 出典: 日本年金機構「令和 7 年 4 月分からの年金額等について」 <https://www.nenkin.go.jp/oshirase/taisetu/kojin/2025/202504/0401.html>
  - 出典: ニッセイ基礎研究所「2025 年度の年金額は 1.9％の増額」 <https://www.nli-research.co.jp/report/detail/id=83880>
- **標準（2026 年度 / 令和 8 年度）**: 2025 年度から **+1.9%** 改定（公表済み）。年額 **847,300 円前後** = 月額 **70,608 円 ≈ 7.06 万円/月**。
  - 出典: 日本年金機構「令和 8 年 4 月分からの年金額等について」 <https://www.nenkin.go.jp/oshirase/taisetu/kojin/2026/202604/0401.html>
  - 出典: 厚生労働省報道発表「令和 8 年度の年金額改定について」 <https://www.mhlw.go.jp/content/12502000/001639615.pdf>
  - 出典: 公的保険アドバイザー協会「2026 年度（令和 8 年度）の年金額が決定しました」 <https://siaa.or.jp/column/175>
- **本コード**: `KOKUMIN_FULL_MONTHLY = 6.8`（**2024 年度値、コメントも「2024 年度水準」と明記**）。
- **判定**: 監査日（2026-04-24）時点で既に **2 年分のスライド改定（合計 +3.84%）が反映されていない**。UI 表示 (`index.html:3348`「2024年度水準」) で明示はあるが、コードの注記だけでは試算値をそのまま `retPensionMonthly` に流し込んだ際に過少になる。
  - 検算: 2026 年度 7.06 万円に対し本コード 6.80 万円 → 過少率 `(7.06 − 6.80) / 7.06 ≈ 0.0368 ≈ 3.68%`（満額時）。
  - ただし長期シミュレーションではマクロスライド `pensionSlide`（0〜20%）で別途調整する前提なので、**3.68% の過少は `pensionSlide` の誤差範囲内**と解釈もできる（§6 で Important）。

### 3.2 国民年金の按分式（加入月数 / 480）

- **標準**: 老齢基礎年金の計算式 = `満額 × (保険料納付済月数 + 免除月数 × 減額係数) / 480`。完全納付前提なら `満額 × 納付月数 / 480`。免除期間がある場合の減額係数（全額免除 1/2、3/4 免除 5/8、半額免除 3/4、1/4 免除 7/8）は考慮されない。
  - 出典: 日本年金機構「老齢基礎年金の計算方法」 <https://www.nenkin.go.jp/service/jukyu/roureinenkin/jukyu-yoken/20150401.html>
- **本コード**: `KOKUMIN_FULL_MONTHLY × (min(years×12, 480) / 480)`。
- **判定**: ✅ 完全納付ケースは式と一致。免除期間がある人は UI に入力欄がないため過大 or 過少（§6 `04-M01`）。

### 3.3 厚生年金（報酬比例部分）の乗率

- **標準**: 老齢厚生年金の報酬比例部分は **期間を 2 つに分けて合算**:
  - 2003 年 3 月以前（総報酬制導入前）: `平均標準報酬月額 × 7.125/1000 × 加入月数`
  - 2003 年 4 月以降（総報酬制導入後、賞与込みベース）: `平均標準報酬額 × 5.481/1000 × 加入月数`
  - 出典: 日本年金機構「報酬比例部分」 <https://www.nenkin.go.jp/service/yougo/hagyo/hoshuhirei.html>
  - 出典: 厚生労働省「年金制度の仕組みと考え方 第 3 公的年金制度の体系（年金給付）」 <https://www.mhlw.go.jp/stf/nenkin_shikumi_003.html>
  - 出典: くらしすと「老齢厚生年金の計算」 <https://www.kurassist.jp/kurassist-eye/nenkin/learn-vol22-20240115.html>
- **本コード**: 5.481/1000 のみ使用。**7.125/1000 区間なし**。
- **判定**: ⚠️ 2003 年 3 月以前の加入期間（2026 年時点で 23 年以上前）を持つ人は **22 歳就職 & 現在 46 歳以上**のケース。試算対象が「現役世代の将来受給見込み」なら 1984 年以前生まれ（= 2026 年時点で 42 歳以上）に有意な誤差が出る。
  - 検算（1980 年生、22 歳就職 = 2002 年 4 月、65 歳退職 = 2045 年、**2003 年 3 月以前の加入は 12 ヶ月**、以降は `43 × 12 − 12 = 504 ヶ月`、平均標準報酬月額 40 万円）:
    - 正しい: `40 × 7.125/1000 × 12 + 40 × 5.481/1000 × 504 = 3.42 + 110.497 = 113.92 万円/年 → 9.49 万円/月`
    - 本コード（5.481 一律・加入 516 ヶ月上限なし）: `40 × 5.481/1000 × 516 = 113.13 万円/年 → 9.43 万円/月`
    - 差: `(9.49 − 9.43) / 9.49 ≈ 0.63%` 過少。1 年分だけ旧乗率が混ざる程度なら誤差は微少。
  - 検算（1970 年生、22 歳就職 = 1992 年 4 月、65 歳退職 = 2035 年、**2003 年 3 月以前の加入は 132 ヶ月**、以降は `516 − 132 = 384 ヶ月`、同年収）:
    - 正しい: `40 × 7.125/1000 × 132 + 40 × 5.481/1000 × 384 = 37.62 + 84.19 = 121.81 万円/年 → 10.15 万円/月`
    - 本コード: `40 × 5.481/1000 × 516 = 113.13 万円/年 → 9.43 万円/月`
    - 差: `(10.15 − 9.43) / 10.15 ≈ 7.10%` 過少。
  - ユーザー層が主に 30〜40 代（= 2003 年時点で学生〜20 歳前後 = 旧乗率期間がほぼゼロ）なら誤差は 1% 未満だが、UI には「加入年数」しかなく生年による自動切替ができない（§6 `04-I01`）。

### 3.4 標準報酬月額の上限

- **標準（2026 年時点）**: 厚生年金の標準報酬月額は 32 等級制、**上限は 65 万円**（月額報酬 63.5 万円以上）。2027 年 9 月から **68 万円**、2028 年 9 月に 71 万円、2029 年 9 月に 75 万円へ段階的引上げが決定済み。
  - 出典: 日本年金機構「厚生年金保険の保険料」 <https://www.nenkin.go.jp/service/kounen/hokenryo/hoshu/20150515-01.html>
  - 出典: 厚生労働省「厚生年金等の標準報酬月額の上限の段階的引上げについて」 <https://www.mhlw.go.jp/stf/seisakunitsuite/bunya/0000147284_00024.html>
- **本コード**: `Math.min(avgIncome/12, 65)`。65 万円/月でキャップ。
- **判定**: ✅ 2026 年時点は一致。ただし段階的引上げは未反映（§6 `04-M02`）。
- **副次判定**: 下限（1 等級 88,000 円 = 8.8 万円/月）のチェックは **なし**。ただし低所得ほど標準報酬月額が下限に張り付くので、低所得層は過大推計になる可能性（§6 `04-M03`）。

### 3.5 標準報酬月額 ≒ 年収 / 12 の扱い

- **標準**: 厚生年金の正式な計算では「**標準報酬月額**（月給の等級別値）」と「**標準賞与額**（賞与の等級別値）」を分けて管理し、その両方を含めた総報酬を加入月数で割った値が **平均標準報酬額**。
  - すなわち年収が賞与込みの「税込年額」なら、`年収 / 12` がそのまま平均標準報酬額に近い近似となる（ただし等級制のため、実際は境界値で丸められる）。
- **本コード**: `avgIncome / 12` を平均標準報酬月額として扱う。UI ラベル「平均年収（税込み）」は賞与込みの想定 → `/12` で賞与按分後の月額相当に変換されるため、**近似としては合理的**。
- **判定**: ✅ 近似として妥当。厳密には等級単位丸め（上下 1 等級まで約 1〜2 万円の差）が生じるが、監査対象のレイヤーでは許容範囲。

### 3.6 繰上げ・繰下げ率

- **標準（2022 年 4 月改正後）**:
  - 繰上げ: **1 ヶ月あたり −0.4%**、最大 60 歳で **−24%**（65 歳との差 60 ヶ月）
  - 繰下げ: **1 ヶ月あたり +0.7%**、最大 75 歳で **+84%**（65 歳との差 120 ヶ月）
  - 昭和 37 年 4 月 1 日以前生まれは旧率（−0.5%/月、最大 30%）
  - 出典: 日本年金機構「年金の繰上げ受給」 <https://www.nenkin.go.jp/service/jukyu/seido/roureinenkin/kuriage-kurisage/20140421-01.html>
  - 出典: 日本年金機構「年金の繰下げ受給」 <https://www.nenkin.go.jp/service/jukyu/seido/roureinenkin/kuriage-kurisage/20140421-02.html>
- **本コード** (`index.html:17815-17818`): `months × 0.004`（繰上）/ `months × 0.007`（繰下）。
- **判定**: ✅ 完全一致。ただし旧率適用対象者（1962 年 4 月 1 日以前生まれ = 2026 年時点で 64 歳超）はカバーしない。これは 2026 年時点で既に 65 歳受給年齢の人々なので、現役世代向けシミュレーションとしては問題なし。
- **ただし**: この率は **表示専用**で `_calcPensionCore` や統合シミュレーションの `state.retirement.pensionMonthly` に反映されない。ユーザーが `retPensionAge` に 70 を入れても、`retPensionMonthly` はユーザーが自分で `×1.42` 倍して入力しないと反映されない（§6 `04-I02`）。

### 3.7 手取り率（約 87%）

- **標準**: 公的年金に対する税・社会保険料は複数要素を合算する:
  - 所得税（雑所得、公的年金等控除後）
  - 住民税
  - 国民健康保険料 / 後期高齢者医療保険料
  - 介護保険料
  - 出典: 国税庁「公的年金等の課税関係」 <https://www.nta.go.jp/taxes/shiraberu/taxanswer/shotoku/1600.htm>
- 一般的な単身高齢者（年金収入 200〜300 万円）の手取り率: **85〜92%**（年金額と世帯構成で変動）。
  - 出典: 金融財政事情研究会「年金生活者の社会保険料負担」 <https://www.kinzai.jp/seminar/> （一般論として引用）
  - 出典: ファイナンシャルフィールド「年金はどれくらい税金や社会保険料がひかれる？」 <https://financial-field.com/pension/entry-179353>
- **本コード**: 一律 `× 0.87`。
- **判定**: ⚠️ 単身者・年金額 200 万円程度では概ね 85〜90% の範囲に入り、0.87 は中央値として妥当。ただし:
  - **低所得（年金額 100 万円未満）**: 非課税世帯となり所得税・住民税はゼロ、社会保険料も軽減 → 手取り率 **95% 超** になる。0.87 を掛けると 8% 過少。
  - **高所得（年金額 400 万円超）**: 所得税・住民税・社保で **80% を切る**こともある。0.87 を掛けると過大。
  - 65〜74 歳と 75 歳以降で国民健康保険→後期高齢者医療保険に切り替わり控除額も変動（年齢軸）。本コードは時間軸も無視。
- 近似精度としては「年金額 150〜300 万円帯の中心値」をとっているが、§6 `04-I03` として Important。

### 3.8 マクロ経済スライド（`pensionSlide`）

- **標準**: マクロ経済スライドは物価・賃金変動率から **スライド調整率**（2025 年度は −0.4%）を控除して年金額改定率を決める仕組み。長期的には年金給付水準を約 20% 削減することで財政均衡を目指す設計（2019 年財政検証）。
  - 出典: 厚生労働省「年金制度の仕組みと考え方 第 7 マクロ経済スライドによる給付水準調整期間」 <https://www.mhlw.go.jp/stf/nenkin_shikumi_007.html>
  - 出典: 厚生労働省「令和 6 (2024) 年財政検証結果」 <https://www.mhlw.go.jp/nenkinkenshou/report/r6.html>
  - 出典: ニッセイ基礎研究所「2025 年度の年金額は 1.9％の増額」 <https://www.nli-research.co.jp/report/detail/id=83880>
- **本コード**: UI 選択値 0%/10%/20%/カスタム（0〜30%）。統合シミュレーション内で `pensionAnnual * (1 − pensionSlide)` として **1 回だけ**適用される（`index.html:15342, 17528, 18315, 18574` 等）。
- **判定**: ⚠️ 重要な論点が 2 つ:
  1. **`calcPensionEstimate` の結果には `pensionSlide` が反映されない**: `calcPensionEstimate` が試算した値は現在水準（2024 年度満額 6.8 万円）ベース。ユーザーがこれを `retPensionMonthly` に転記すると、統合シミュレーションで改めて `(1 − pensionSlide)` が掛かる。一見二重にマクロスライドが効きそうに見えるが、UI の文言（「財政検証標準」等）からは**将来の累積削減率**を入れる想定なので、試算値が現在水準なら一回だけの適用が正しい。ユーザーが理解して使う前提は成立しうる。
  2. **一括適用は長期効果の近似としてのみ妥当**: 本来マクロスライドは**毎年**の改定率調整で、受給開始以降の年金額にも適用される（既裁定者へのスライドが 2018 年度以降は現役との連動になった）。本コードは「受給開始時点から一律 N% 減」というスナップショット表現。長期的な受給総額が少なくなる近似としては妥当だが、受給 20 年間の途中で減額が進行する現実とは異なる。
- §6 `04-I04` として Important 記録。

### 3.9 第 3 号被保険者の扱い（パートナー年金）

- **標準**: 第 2 号被保険者（厚生年金加入者）の配偶者で、かつ年収 130 万円未満の者は **第 3 号被保険者** となり、国民年金保険料を自己負担せずに満額の基礎年金を受給できる。厚生年金は受給できない。
  - 出典: 日本年金機構「第 3 号被保険者」 <https://www.nenkin.go.jp/service/kokunen/kanyu/20140710-04.html>
- **本コード**: パートナー用 UI (`index.html:3372-3420`) は本人と同じフォーム（雇用形態セレクト含む）。
  - パートナーが `employType = 'employee'`（会社員）なら厚生年金あり（自分で働いている想定）
  - パートナーが `employType = 'self'` 自営業なら国民年金のみ
  - パートナーが `employType = 'part'` パート → 厚生年金加入なし
  - **第 3 号被保険者（専業主婦/主夫など）の選択肢が UI にない**。`'part'` か `'self'` で代替するのが近そう（どちらも厚生年金なし、国民年金 40 年満額）。
- **判定**: ⚠️ 第 3 号を明示できないが、`'part'` でも `'self'` でも国民年金のみの計算になるため、**計算結果としては第 3 号と同じ**（パートは月 8.8 万円超で第 2 号になる条件があるが UI は意識しない）。UI/UX の欠陥ではあるが計算結果には影響しない。§6 `04-M04` として Minor。

### 3.10 加入月数と就業開始年齢の扱い

- **標準**: 厚生年金加入月数 = 就職月〜退職月（最大 60 歳/70 歳）、国民年金は 20〜60 歳の 40 年。
- **本コード**: 国民年金 `kokuminYears` と厚生年金 `koseiYears` を**独立に入力**させる設計。
- **判定**: ✅ 実態に沿う（転職で厚生年金→国民年金→厚生年金と変遷するケースも個別入力で表現可能）。ただし:
  - 20〜22 歳の学生期間は国民年金加入義務（学生納付特例で猶予可能）。UI で「22 歳就職・厚生年金 43 年」と入れると国民年金 `40` のままだと両方加算され二重カウントにならないか確認: `kokuminMonthly` と `koseiMonthly` は**独立に計算され合算**されるので、国民年金 40 年は基礎年金満額 6.8 万円、厚生年金 43 年は報酬比例部分。**両者の重複はなく正しい**（基礎年金と厚生年金は独立）。
  - ただし被保険者種別の重複（厚生年金加入中の人は自動的に第 2 号被保険者 = 国民年金加入者なので、国民年金加入年数 = 厚生年金加入年数 + 学生特例納付分 + 自営業期間等）の関係が UI で説明されず、ユーザーが混乱する余地あり（§6 `04-M05`）。

### 3.11 在職老齢年金（61 歳以降の就労時の減額）

- **標準**: 65 歳以上で給与と厚生年金の合計が **月 51 万円（2026 年度）** を超えると超過分の半額が年金から減額される。2027 年度から **月 65 万円**に引上げ予定。
  - 出典: 日本年金機構「在職老齢年金」 <https://www.nenkin.go.jp/service/jukyu/roureinenkin/zaishokurourei/zaishoku.html>
  - 出典: 社会保険労務士法人 栗木会計事務所「令和 8 年度の年金額改定」 <https://www.syakaihoken.jp/17691305375172>
- **本コード**: 在職老齢年金による減額は **一切実装なし**。`calcPensionEstimate` はあくまで「65 歳退職前提」での満額試算。
- **判定**: ⚠️ 退職再雇用で 65 歳以降も働く人には過大推計。ただし UI は「退職後の年金」を試算する立ち位置なので、ユーザーが 65 歳退職なら問題なし。§6 `04-M06` として Minor。

## 4. 仮定・制約

1. **満額基準が 2024 年度値で固定**: `KOKUMIN_FULL_MONTHLY = 6.8` は令和 6 年度（2024 年度）値。2026 年度値 7.06 万円への更新が必要。
2. **厚生年金の 2003 年 3 月以前区間は無視**: 乗率 7.125/1000 の期間はすべて 5.481/1000 として計算される。現役 30 代では影響ほぼなし、40 代後半〜50 代で数 % の過少。
3. **免除期間・学生納付特例期間は未対応**: 加入年数のみ入力で、免除区分による減額係数（1/2〜7/8）は反映不可。
4. **標準報酬月額の等級丸めは無視**: `avgIncome/12` の連続値をそのまま使用。
5. **標準報酬月額上限の将来引上げ（68/71/75 万円）は反映なし**: 2026 年時点の 65 万円のみ。
6. **標準報酬月額下限（8.8 万円）チェックなし**: 年収 100 万円未満でも `avgIncome/12` がそのまま使われる。
7. **手取り率は一律 87%**: 年金額・年齢・世帯構成で実際は 80〜95% の幅があるが近似値固定。
8. **繰上げ・繰下げ率は表示のみ**: `_calcPensionCore` の出力に反映されず、ユーザーが手動で `retPensionMonthly` を書換える想定。
9. **マクロ経済スライドは統合シミュで一括適用**: `calcPensionEstimate` の試算値には適用されず、ユーザーの入力と `pensionSlide` の 2 段階。
10. **第 3 号被保険者の明示選択なし**: 主婦/主夫は `'part'` / `'self'` で代替。計算結果は同じ。
11. **在職老齢年金の減額未対応**: 65 歳以降の就労併用時に過大推計。
12. **配偶者加給年金・振替加算は未対応**: 年上の厚生年金加入者に対する配偶者加給年金（年額 22〜39 万円程度、昭和 18 年 4 月 2 日以降生まれなら特別加算あり）や、それが配偶者の基礎年金に振替わる振替加算は一切計算されない。
13. **障害年金・遺族年金は対象外**: 老齢年金のみ。

## 5. エッジケース

1. **`kokuminYears = 0`**: `kokuminMonths = 0` → `kokuminMonthly = 0`。正しい（未加入）。
2. **`kokuminYears = 41`（UI 上限 40 のはずだが直接入力で超過）**: `min(41×12, 480) = 480` で 40 年扱い。クランプ正しい。
3. **`avgIncome = 0` かつ `employType = 'employee'`**: 厚生年金計算スキップ → `koseiMonthly = 0`。正しい。
4. **`koseiYears = 46`（UI 上限 45 超）**: 46×12 = 552 ヶ月で計算される。厚生年金は通常 70 歳まで（最大 48 年程度）加入可能なので現実離れはしないが、80 歳まで加入のような非現実ケースのクランプはない。
5. **`employType = 'self'` or `'part'` + `avgIncome` 設定**: `if (employType === 'employee' ...)` 分岐でスキップされるため `koseiMonthly = 0`。正しい。
6. **`avgIncome = 1000`（年収 1000 万円）**: `hyojunGekkyu = min(83.33, 65) = 65`。上限で正しくクランプ。
7. **`avgIncome = 50`（年収 50 万円 = 低所得）**: `hyojunGekkyu = 50/12 = 4.17 万円`。標準報酬月額下限 8.8 万円（1 等級）より下だが、コードは下限チェックなしで 4.17 でそのまま計算 → 過少。§6 `04-M03`。
8. **`avgIncome` 負値**: UI `min="0"` で防御されるが、直接入力で負値入力された場合 `/12` も負 → 厚生年金が負値になる。`Math.min(負値, 65) = 負値` のまま通過。**防御なし**（§6 `04-M07`）。
9. **複数回実行**: `window._pensionEstimate` に保存されるため副作用ありだが、`_calcPensionCore` 自体は純粋関数で idempotent。
10. **本人 `single` & `partner` タブ開いて入力**: `hasParter = (marital === 'married' || marital === 'planning')` で `single` の場合 `partner` の値は世帯合計に含めない。正しい。
11. **`marital = 'planning'` で `partner` 未入力**: `window._pensionEstimate.partner = undefined` → `partnerNet = 0`。`selfNet + 0 > 0` ならセルフのみ表示。正しい。
12. **旧 `calcPensionEstimate()` 呼び出し（`calcSimpleSim`・`renderTaxSimulation`）**: 引数なしで呼ばれると `who = undefined → 'self'` となり、本人の DOM 要素を探して描画しようとする。DOM が存在しない文脈で呼ばれると `document.getElementById('pensionEstimateResult')` が `null` で `.style.display = ''` でエラー。防御はコード内にない。§6 `04-C01`。

### 🟡 Important

- **`04-I01` 厚生年金の 2003 年 3 月以前区間（乗率 7.125/1000）が未対応**
  - §3.3 参照。40 代後半以上のユーザーに最大 7% 前後の過少推計が出る。
  - 数値インパクト（1970 年生・平均年収 500 万・65 歳退職）:
    - 正しい: `41.67 × 7.125/1000 × 132 + 41.67 × 5.481/1000 × 384 = 39.19 + 87.70 = 126.89 万円/年 → 10.57 万円/月`
    - 本コード（5.481 一律）: `41.67 × 5.481/1000 × 516 = 117.84 万円/年 → 9.82 万円/月`
    - 過少: `(10.57 − 9.82) / 10.57 ≈ 0.0710 ≈ 7.10%`
  - **修正方針**: 生年月日（`state.profile.birth`）から 2003 年 3 月時点の年齢を逆算し、「2003 年 3 月以前の加入月数」を推定。または UI に「2003 年 3 月以前の加入年数」欄を追加。

- **`04-I02` 繰上げ・繰下げ率が試算結果に反映されない（計算は正しいが表示専用）**
  - `calcAndShowDeferralSim` は 60〜75 歳の月額テーブルを表示するだけ。`applyPensionEstimate` は 65 歳基準値をそのまま `retPensionMonthly` に転記する。
  - UI の `retPensionAge`（受給開始年齢）は統合シミュレーションで **年金受給の開始年を遅らせるだけ**で、月額は変動しない。繰下げで 70 歳受給を選んだユーザーは、月額は 65 歳基準のまま・受給開始だけ 70 歳に遅れる = **金額増額が効かない = シミュレーションで過少推計**。
  - 数値インパクト（65 歳基準月 15 万、70 歳繰下げ、余命 90 歳）:
    - 正しい: 70 歳から月 `15 × 1.42 = 21.3 万円` × 20 年 = `21.3 × 12 × 20 = 5,112 万円`
    - 本コード: 70 歳から月 `15 万円` × 20 年 = `15 × 12 × 20 = 3,600 万円`
    - 過少: `(5112 − 3600) / 5112 ≈ 0.2958 ≈ 29.58%`（繰下げ分の総受給増が完全欠落）
  - **修正方針**: `applyPensionEstimate` で `retPensionAge` が 65 と異なる場合、`_calcPensionCore` の出力に `adjustRate(retPensionAge)` を掛けて反映。または統合シミュレーション側で `pensionMonthly` を `retPensionAge` に応じて自動補正。

- **`04-I03` 手取り率 0.87 固定は年金額・年齢で大きくぶれる**
  - §3.7 参照。
  - 数値インパクト（試算 3 パターン）:
    - 年金額 100 万円/年（低所得・非課税世帯）: 実際の手取り率 ≈ 0.97 → 本コード 0.87 × 100 = 87 万円、実際 97 万円 → 11% 過少。
    - 年金額 200 万円/年（中位）: 実際 0.88 → 0.87 で誤差 −1%。OK。
    - 年金額 400 万円/年（高所得）: 実際 0.80 程度 → 0.87 で 8% 過大。
  - **修正方針**: 年金額階層別の手取り率テーブルを用意（100 万未満 95%、100〜200 万 90%、200〜300 万 87%、300〜400 万 83%、400 万超 78% 等）。または年齢（65〜74 / 75〜）で国保・後期高齢者の区分を追加。

- **`04-I04` `calcPensionEstimate` の結果と `pensionSlide` の二重適用の懸念**
  - `calcPensionEstimate` は 2024 年度満額 6.8 万円ベース（§3.1）で計算し、それをユーザーが `retPensionMonthly` にコピー → 統合シミュレーションで `× (1 − pensionSlide)` を掛ける。
  - UI 文言では `pensionSlide = 10%` を「財政検証標準」としており、2024 年度値に対して **将来累積 10% 減**を掛ける想定。現在水準ベースで計算した値に将来削減率を掛けるのは**設計通り**だが、UI にその関係性が明示されていない。
  - 混乱の余地:
    1. `calcPensionEstimate` 表示で「2024 年度水準」と明記 → ユーザーが「まだ削減されていない」と理解
    2. `retPensionMonthly` にコピー
    3. 統合シミュレーションで `pensionSlide = 10%` で −10% 適用 → 最終的に 2024 年度 × 0.9
    4. しかし 2026 年度満額は既に +3.8% → 正しい将来値は `2026 年度値 × (1 − 残りスライド)` であり、+3.8% 上昇を無視して −10% だけ掛けると合計 13.8% 過少推計の可能性。
  - 数値インパクト（基礎年金満額 40 年加入、2026 年度基準、将来 10% 削減）:
    - 正しい: `7.06 × (1 − 0.10) = 6.35 万円/月`
    - 本コード: `6.80 × (1 − 0.10) = 6.12 万円/月`
    - 過少: `(6.35 − 6.12) / 6.35 ≈ 0.0362 ≈ 3.62%`
  - **修正方針**: `KOKUMIN_FULL_MONTHLY` を毎年度更新するスクリプト/プロセスを整備、または UI 文言を「現在年度（X 年度）水準」と動的表示し、`pensionSlide` の基準年を明記。

### 🟢 Minor

- **`04-M01` 国民年金の免除期間の減額係数が未対応**
  - UI には加入年数のみの入力で、全額免除（減額係数 1/2）、3/4 免除（5/8）、半額免除（3/4）、1/4 免除（7/8）の区別は不可。学生納付特例や産前産後免除も同様。
  - 影響は学生時代に免除だった人（多くの社会人が該当）に限定され、40 年満額ではなく 38 年相当になる → 5% 程度の過大推計（免除→未納扱い時の補正）。
  - **修正方針**: 「免除年数」入力欄を追加し、係数 1/2 で加算（全額免除想定）。

- **`04-M02` 標準報酬月額上限の将来引上げ（2027 年 9 月〜）が未反映**
  - 65 → 68（2027.9）→ 71（2028.9）→ 75（2029.9）万円の段階的引上げが決定済み。
  - 影響は年収 800 万円超の高所得層で、将来の厚生年金試算が過少。
  - 数値インパクト（年収 1000 万・2030 年退職）:
    - 正しい（最終 75 万円上限）: 部分的に 75 万円で計算されるため、65 万円固定より約 15% 上。
    - 本コード: 65 万円で固定。
  - **修正方針**: 上限値を年度別テーブルにし、退職年齢と加入月数で時系列重み付け。

- **`04-M03` 標準報酬月額下限（8.8 万円）チェックなし**
  - 年収 50 万円（月 4.17 万円）のような低所得で `hyojunGekkyu = 4.17` がそのまま使われる。正しくは 1 等級 8.8 万円で下限クランプされるべき（実際は厚生年金加入できない月 8.8 万未満の給与もあるが、そもそもその場合は非加入）。
  - 逆方向の過小推計（4.17 < 8.8 なので下限を適用すると厚生年金月額が増える）。ただし、月給 8.8 万未満は厚生年金非加入が実態なので、実際に影響するケースは少ない。
  - **修正方針**: `hyojunGekkyu = max(min(avgIncome/12, 65), 8.8)` と両面クランプ。

- **`04-M04` 第 3 号被保険者の明示的な選択肢がない**
  - §3.9 参照。`'part'` か `'self'` で代替可能で計算結果は同じ。UI/UX のみの問題。
  - **修正方針**: `employType` に `'housewife'` を追加し、「専業主婦/主夫（第 3 号被保険者）」ラベル。

- **`04-M05` 国民年金加入年数と厚生年金加入年数の関係説明なし**
  - 厚生年金加入中は同時に第 2 号被保険者として国民年金にも加入している。UI 上は独立入力のため、ユーザーが「厚生年金 40 年 + 国民年金 40 年」と入力しても基礎年金は 40 年満額扱いで問題ないが、論理的な整合性説明が UI にない。
  - **修正方針**: UI に注記「厚生年金加入期間は国民年金加入期間にも含まれます（自動合算）」。

- **`04-M06` 在職老齢年金による減額（61 歳以降就労時）が未対応**
  - §3.11 参照。2026 年度基準は月 51 万円超で減額、2027 年度からは 65 万円超。
  - 影響は 65 歳以降も就労継続する層に限定。`calcPensionEstimate` の利用シーン（将来の年金見込み試算）では 65 歳退職前提が多いため影響軽微。
  - **修正方針**: `retirement` に「65 歳以降の就労月収」フィールドを追加し、月額合計 51 万円（または年度別基準）超過分の半額を差し引く。

- **`04-M07` `avgIncome` 負値の防御なし**
  - UI の `min="0"` は HTML バリデーションで回避されるが、直接入力や既存データが負値だった場合、`_calcPensionCore` は負の厚生年金月額を出す。
  - **修正方針**: `const hyojunGekkyu = Math.max(0, Math.min(avgIncome/12, 65));`

- **`04-M08` 配偶者加給年金・振替加算が未対応**
  - 年上の厚生年金加入者に 20 年以上の加入と年下配偶者がいる場合、配偶者加給年金（年 22〜39 万円）が付く。65 歳以降は配偶者の基礎年金に振替加算として移る。
  - 影響は該当世帯で月 2〜3 万円のズレ。
  - **修正方針**: 配偶者年齢差・厚生年金加入年数が閾値超なら加給年金を自動加算するロジックを追加。要件複雑なためオプション機能扱いが妥当。

- **`04-M09` 令和 8 年度（2026 年度）満額値への定数更新**
  - §3.1 参照。`KOKUMIN_FULL_MONTHLY = 6.8`（2024 年度値）→ 2026 年度 7.06 万円。
  - コード `index.html:15098-15100` のコメント「2024 年度水準」も更新要。
  - **修正方針**: `KOKUMIN_FULL_MONTHLY = 7.06` に更新、コメントも差し替え。ただし `04-I04` で述べた通り `pensionSlide` の基準年も連動で見直しが必要。年度更新タスクとしてメンテナンス枠で管理。

## 7. 結論

- **この領域の信頼度: ❌ 要対応**
- **判定根拠**: `04-C01` は `calcSimpleSim`（かんたん試算）と `renderTaxSimulation`（税シミュレーション表示）の 2 つのユーザー可視機能で `pension = undefined` を発生させる実バグ。「メイン（`calcIntegratedSim`）は影響なし」は部分的には正しいが、上記 2 機能が UI から起動できる限り、安全側で ❌ 要対応 とする。加えて `04-I02`（繰下げ未反映で 70 歳繰下げ時に約 30% 過少）、`04-I01`（2003 年 3 月以前区間未対応）、`04-I03`（手取り率 0.87 固定でエッジ誤差）、`04-I04`（マクロスライド二重適用懸念）が重なるため Important 以上の修正が必要。
- **一言サマリー**: コア式は正しい方向性（`5.481/1000`、`65 万円上限`、`0.4% / 0.7%` 繰上繰下率）を押さえている実務的な近似計算だが、**`04-C01` の重複定義による `calcSimpleSim` 破損**、**`04-I02` の繰下げ率が試算に反映されないこと**、**`04-I01` の 2003 年 3 月以前区間未対応**、**`04-I03` / `04-I04` の手取り率・満額基準の古さ**で、中年層以降のユーザーには 5〜30% のズレが生じる。他領域（ⓕ ２プール、ⓐ 統合シム）の監査で `pensionSlide` や `retPensionAge` の最終適用ロジックを確認すべき。
