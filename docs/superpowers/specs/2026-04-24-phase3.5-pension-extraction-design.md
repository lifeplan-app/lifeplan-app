## 1. 背景と目的

Phase 3 では計算ロジック 約 20 関数を `calc/` 下の 8 ファイルに抽出した。Phase 4（Important 43 件の修正）を進めるにあたり、**年金コア関数 `_calcPensionCore` が index.html に残っている**ことが修正作業の局所化を妨げる。

`_calcPensionCore` は：

- 完全な純粋関数（DOM 非依存）
- Phase 2 監査レポート（`docs/phase2-audits/04-pension.md`）で監査済み
- Phase 4 の `04-I02`（繰下げ率が試算に反映されない）修正時に触る可能性が高い

したがって Phase 3 のスコープ外扱いだったこの 1 関数だけを、**最小限の追加フェーズ**として抽出する。

本フェーズの目的は Phase 4 の作業局所化であり、**挙動は一切変えない**（Phase 1 snapshot 差分ゼロが成功基準）。

## 3. アーキテクチャ

### 3.1 Before / After

```
【Before】
index.html:
  function _calcPensionCore(...)        ← 純粋関数
  function calcPensionEstimate(who)     ← DOM glue
  function calcSimplePensionEstimate()  ← DOM glue

【After】
calc/pension.js:                         ← 新規（約 20 行）
  function _calcPensionCore(...)

index.html:
  <script src="calc/pension.js"></script>  ← 既存 script タグ群に追加
  function calcPensionEstimate(who)     ← 残る（ブラウザの global scope 共有により _calcPensionCore を参照可能）
  function calcSimplePensionEstimate()  ← 残る
```

### 3.2 ロード順序

`<script src="calc/retirement.js">` の**後**に `<script src="calc/pension.js">` を追加する。理由：

- 現時点では `_calcPensionCore` と `calc/retirement.js` の間に依存関係はない
- しかし Phase 4 `04-I02` 修正で `calcRetirementSimWithOpts` が `_calcPensionCore` を呼ぶ拡張になる可能性がある
- その場合、先に `retirement.js` が定義されている方が読みやすい配置（ただし classic script の関数宣言は hoist されるため実行時問題はない）

実際の配置は `<script src="calc/integrated.js">` の後でも機能的には同じ。ここでは「retirement の直後」とする。

## 8. Phase 4 への橋渡し

Phase 4 で `04-I02`（年金繰下げ反映）を修正する時、以下のいずれかの経路を選べる：

- **経路 A**: `calc/pension.js` の `_calcPensionCore` に繰下げ率引数を追加
  - 新 signature: `_calcPensionCore(employType, koseiYears, avgIncome, kokuminYears, pensionAge = 65)`
  - 繰下げ率 `adjustRate(pensionAge) = pensionAge > 65 ? 1 + (pensionAge - 65) * 12 * 0.007 : 繰上/繰下計算` を適用
- **経路 B**: `calc/retirement.js` の `calcRetirementSimWithOpts` で年金収入に繰下げ率を掛ける
  - `pensionAnnual *= adjustRate(pensionAge)` を追加
- **経路 C**: UI の繰下げ選択時に `state.retirement.pensionMonthly` を直接書き換える（UI 経路）

Phase 3.5 で `_calcPensionCore` が独立ファイルになれば、どの経路でも修正箇所が明確。

## 9. スコープの正当化

「`_calcPensionCore` だけ抽出する Phase 3.5 って小さすぎない？」という疑問への回答：

- 13 行の関数抽出であっても、**抽出自体は Phase 3 の 9 回のサイクルと同じ手順**（1 つのタスクで完結する分、管理コストは低い）
- Phase 4 の `04-I02` 修正時、抽出と修正を混ぜるよりも **Phase 3.5 で抽出 → Phase 4 で修正** と分離する方が、変更履歴の理解・ロールバックが容易
- Phase 2 監査で「`_calcPensionCore` は純粋関数」と明記されているので、抽出 = 読み替えのみで済む
- 将来、税計算や相続・贈与等の追加抽出が必要になった場合、Phase 3.5 と同じパターン（最小スコープで抽出）で進める前例になる
