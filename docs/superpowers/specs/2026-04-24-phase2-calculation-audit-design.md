# Phase 2｜計算ロジック監査 設計書

- **作成日**: 2026-04-24
- **対象アプリ**: ライフプランアプリ（`index.html`）
- **位置付け**: 計算ロジックの正確性検証 3段階プランの **Phase 2**
- **前提フェーズ**: Phase 1（スナップショット・テスト）完了済み（コミット `1f83582`）
- **次のフェーズ**: Phase 3（計算ロジックの `index.html` からの分離）

---

## 1. 背景と目的

### 1.1 Phase 1 で達成したこと

- 本物の `index.html` をヘッドレスブラウザ（Playwright）で実行し、5シナリオ分の年次出力をスナップショット化する安全網を構築
- 「**今の計算結果が意図せず変わる**」ことは自動検知できるようになった

### 1.2 Phase 1 で未達成だったこと

- 「**今の計算結果が正しいか**」は依然として未検証
- スナップショットは現状を固定するだけで、現状が正しいとは保証しない

### 1.3 Phase 2 の目的

**「計算ロジックが FP実務として妥当か」を人間が読める監査レポートで明らかにする。**

完全一致を求めず、以下2つを明確にする：

- 各主要計算単位が使っている計算式が、FP教科書・公式計算・税法等の標準と一致しているか
- 年次キャッシュフローの動きが、典型的な家族像（シナリオB）で筋の通った結果を返しているか

**計算ロジックの修正は Phase 2 のスコープ外**。見つかった問題は優先度付きリストとして整理するが、修正は別フェーズで行う。

---

## 2. アプローチ

独立参照実装（Python/Excel）は行わない。理由：

- 完全一致を求めていないため、独立実装のコストに見合わない
- 非エンジニアのユーザーが保守できない
- 新実装のバグが検証ノイズになるリスク

代わりに採用するアプローチ：

- **ロジック監査**: 各領域のコードを読み、計算式を FP標準と突き合わせ、仮定・エッジケース・問題を整理した監査レポートを作成
- **サニティウォークスルー**: 1シナリオ（シナリオB）について年次出力を人間が読める日本語で解説し、動きが筋が通るかを判定
- **問題集約**: 全領域の問題を優先度付きリストに集約

## 4. 監査の順序

依存関係（下位が上位を呼ぶ）に沿って基盤から統合へ：

```
基盤
 1. ⓖ 資産成長複利 + 税引き
 2. ⓗ 収入・支出の経年変化
 3. ⓘ ライフイベント費用

構成部品
 4. ⓒ 年金試算
 5. ⓓ 住宅ローン + 控除
 6. ⓔ パートナーリタイア
 7. ⓕ 二プールモデル

統合・終端
 8. ⓑ 出口戦略・4プール取り崩し
 9. ⓐ 統合キャッシュフロー

最後
 ・サニティウォークスルー（シナリオB）
 ・summary-and-issues.md
```

**この順序のメリット**：

- 基盤で見つかった問題は上位領域でも再現するはず → 上位監査の信頼性が高まる
- 統合層で「数字がおかしい」となった時、下位監査の結果を根拠に原因領域を絞り込める

## 5. 監査レポートの固定フォーマット

各領域の監査レポートは `docs/phase2-audits/NN-<領域名>.md` に保存。以下の章構成で統一する：

```markdown
# 監査レポート：[領域名]

## 対象範囲
- 対象関数: `fn` (index.html:line), ...
- 呼び出し元: どこから使われているか

## 1. 関数の目的と入出力
- 何を計算するか
- 引数・戻り値

## 2. 使用している計算式
- 計算式を数式で明示
- コードからの該当箇所引用

## 3. 標準との突合
- FP教科書・公式資料・税法等の標準的な計算式と並べて比較
- 一致 / 差異を明記
- 差異があれば「意図した差異か、バグか」の判定

## 4. 仮定・制約
- 暗黙に仮定していること
- 実運用で仮定が崩れる条件

## 5. エッジケース
- 特殊値入力時の挙動
- 実装に防御があるか

## 6. 検出された問題（深刻度付き）
- 🔴 Critical: 明確なバグまたは税法違反
- 🟡 Important: 仮定の説明不足・UI誤解リスク・特殊ケース未対応
- 🟢 Minor: 命名・コメント・軽微な改善

## 7. 結論
- この領域の信頼度: ✅ 高 / ⚠️ 中 / ❌ 要対応
- 一言サマリー
```

領域によって分量は変動するが、**章構成は共通**にすることで他領域との比較・参照性を確保する。

## 6. サニティウォークスルー

### 6.1 対象シナリオ

**シナリオB：鈴木健太 35歳・共働き夫婦・子1人・住宅ローン**

選定理由：

- 9領域すべてが絡む（収入・支出・運用・ローン・ライフイベント・パートナー・退職・年金）
- 現役世代から退職後まで55年超の長期シミュレーションで、年次変化のダイナミクスが見える
- 典型的な日本の家族像で FP実務の「それっぽさ」を判定しやすい
- 他シナリオ（A独身・C FIRE・D 55歳・E シングルマザー）はいずれも特殊

### 6.2 解説方式

全55年を逐年解説するのではなく、**転換点10〜15ポイントを詳述＋間の年は表で数値のみ提示**。

解説対象の年（目安）：

- 初年度
- 子の進学タイミング（小・中・高・大、4点）
- 住宅ローン完済年
- 本人定年退職年
- パートナー定年退職年
- 年金受給開始年（本人・パートナー）
- 資産ピーク年
- 資産枯渇年（もしあれば）
- 最終年（余命時）

### 6.3 レポート構造

```markdown
# サニティウォークスルー：シナリオB 鈴木健太

## 年次ハイライト

### 2026年（初年度・35歳）
- 総収入: 690万円（本人420 + パートナー270）
- 総支出: 432万円（生活費360 + ローン72）
- 資産残高: 530万円（現金200 + 投資330）
- 解説: ...
- 判定: ✅ 妥当 / ⚠️ 違和感あり

### [他の区切り年も同様]

## 区切り年間の年次サマリー（表形式・数値のみ）
| 年 | 年齢 | 収入 | 支出 | 資産残高 |
| ... |

## 検出された違和感
- 「〇〇年目で資産が唐突にジャンプしている→理由が不明」
- ...

## 全体的な所感
- 資産推移の形: ✅/⚠️/❌
- 年金水準: ✅/⚠️/❌
- ローン完済タイミング: ✅/⚠️/❌
- 退職後取り崩しペース: ✅/⚠️/❌
```

## 11. 既知のリスクと対処

| # | リスク | 対処 |
|---|--------|------|
| 1 | 監査の主観性（「標準と一致」の判定が監査者依存） | 標準の参照元（教科書名・公式URL・税法条文）を明記して根拠を残す |
| 2 | Critical が多数見つかった場合の優先順位付け | 影響を受けるシナリオ数・金額インパクト・修正難易度で評価 |
| 3 | サニティウォークスルーで違和感があっても「監査では気付けない暗黙のバグ」の可能性 | 疑わしい転換点は Important 扱いで記録、Phase 3 以降で精査 |
| 4 | FP業界標準が地域・時代で異なる（例：インフレ率前提、退職金計算） | 前提を明示し、監査では「一般的な日本のFP実務」を基準とする |
| 5 | `index.html` が23,000行超で関連コードの横断把握が困難 | 各領域監査の最初に「関連ファンクション一覧」を列挙してスコープを固定 |

## 12. 次フェーズ（Phase 3）への接続

Phase 2 の成果物は Phase 3 でも活用される：

- サニティウォークスルーが、Phase 3 リファクタ前後で挙動が変わっていないかの比較基準になる
