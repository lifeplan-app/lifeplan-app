## 1. 背景と目的

Phase 2 の監査で以下が明らかになった：

- 計算のコア数式（複利・元利均等返済・NISA枠・税率）は FP 実務と整合している

Important / Minor は Phase 3 の過程で併せて修正する方針（Phase 2 summary 記載）。

## 3. 修正ワークフロー

全体の流れは「グループ単位でサイクルを回す」方式：

```
for each group (計6グループ):
  1. docs/phase2-5-fixes/expected-changes.md に「期待される変化方向」を事前宣言
  2. index.html を修正（必要なら UI 要素を追加）
  3. npm test を実行 → Phase 1 スナップショットが赤くなるはず
  4. git diff test/__snapshots__/ を目視確認:
     - 宣言した方向に数値が動いているか
     - 想定外の箇所まで動いていないか
  5. 問題なければ:
     - npm run test:update でスナップショット更新
     - fix + 更新 snap + (UI変更) をまとめて 1 コミット
     - expected-changes.md に「実際の snapshot 差分サマリー」を追記
  6. 問題あれば:
     - 修正を戻し、原因調査、再挑戦
```

**Phase 1 の snapshot 安全網が本来の用途（意図しない変動の検知＋意図した変動の承認）で機能する**ため、追加のテスト基盤は作らない。
